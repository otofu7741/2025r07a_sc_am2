**問題概要**

| 選択肢 | 内容                                                                                                   | 正誤  |
| ------ | ------------------------------------------------------------------------------------------------------ | ----- |
| ア     | 運用監視暗号リスト＝運用監視システムにおける利用実績が十分と判断された推奨暗号                         | ×     |
| イ     | 証明書失効リスト＝政府共用認証局が公開する危殆化した暗号技術のリスト                                   | ×     |
| **ウ** | **推奨候補暗号リスト＝安全性・実装性能が確認され、今後推奨暗号リストに掲載される可能性がある暗号技術** | **○** |
| エ     | 電子政府推奨暗号リスト＝互換性維持目的に限った継続利用を推奨                                           | ×     |

**解説**

### 問2　Pass the Hash攻撃
**正解：イ**

**問題概要**
Pass the Hash攻撃の説明として正しいものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                                   | 正誤                          |
| ------ | ---------------------------------------------------------------------- | ----------------------------- |
| ア     | ハッシュ値から平文パスワードを導出してログイン                         | ×（これはクラック）           |
| **イ** | **パスワードのハッシュ値だけでログインできる仕組みを悪用してログイン** | **○**                         |
| ウ     | パスワードを固定し、IDのハッシュ化を繰り返してログインを試みる         | ×（リバースブルートフォース） |
| エ     | ハッシュ化されずに保存されている平文パスワードを使ってログイン         | ×                             |

**解説**
Pass the Hash（PtH）攻撃は、Windows認証（NTLMなど）において、**パスワードのハッシュ値そのものを認証トークンとして悪用**する攻撃。攻撃者は正規ユーザのメモリやSAMデータベースからNTLMハッシュを奪取し、平文パスワードを復元せずそのまま別システムに対して認証を試みる。

アはパスワードクラック（ハッシュ→平文の逆算）であり別の攻撃。ウはリバースブルートフォース（パスワードスプレー）に近い説明。エは平文パスワードがそのまま流出しているケース。

### 問3　PQC（Post-Quantum Cryptography）
**正解：エ**

**問題概要**
PQC（耐量子暗号）の説明として正しいものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                                                           | 正誤                              |
| ------ | ---------------------------------------------------------------------------------------------- | --------------------------------- |
| ア     | 量子コンピュータで暗号化処理速度を飛躍的に向上させる暗号技術                                   | ×                                 |
| イ     | 量子・従来どちらのコンピュータでも処理時間を一定にできる暗号技術                               | ×                                 |
| ウ     | 量子コンピュータによる攻撃では解読時間が大幅に短くなる可能性のある暗号技術                     | ×（これは既存暗号の脆弱性をいう） |
| **エ** | **量子コンピュータを使った攻撃に対しても従来のコンピュータを使って安全性を確保できる暗号技術** | **○**                             |

**解説**
PQC（Post-Quantum Cryptography：耐量子暗号/ポスト量子暗号）とは、量子コンピュータを使った攻撃（ShorのアルゴリズムによるRSA・ECCへの攻撃、Groverのアルゴリズムによる対称鍵暗号への攻撃）に対して安全性を持つ暗号技術の総称。**量子コンピュータを使って動作させる暗号ではなく、量子コンピュータの攻撃に耐える暗号**である点が重要。格子暗号、符号暗号、多変数多項式暗号などが候補。NISTは2024年に初のPQC標準（ML-KEM、ML-DSA等）を策定した。

### 問4　PKIのRA（Registration Authority）の役割
**正解：エ**

**問題概要**
PKIを構成するRAの役割として正しいものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                               | 正誤                               |
| ------ | ------------------------------------------------------------------ | ---------------------------------- |
| ア     | デジタル証明書にデジタル署名を付与する                             | ×（CAの役割）                      |
| イ     | デジタル証明書に紐付けられた属性証明書を発行する                   | ×（AA：Attribute Authorityの役割） |
| ウ     | デジタル証明書の失効リストを管理し有効性を確認する                 | ×（CRL管理はCAまたはVAの役割）     |
| **エ** | **本人確認を行い、デジタル証明書の発行申請の承認または却下を行う** | **○**                              |

**解説**
PKIの主要コンポーネント：
- **CA（Certificate Authority）**：証明書の発行・署名・失効
- **RA（Registration Authority）**：申請者の本人確認・審査を担当し、CAへ発行承認/却下を通知
- **VA（Validation Authority）**：証明書の有効性確認（OCSPなど）
- **AA（Attribute Authority）**：属性証明書の発行

RAは「受付窓口」的役割で、CAへの直接アクセスを減らしてセキュリティを高める機能もある。

### 問6　XMLデジタル署名の特徴
**正解：イ**

**問題概要**
XMLデジタル署名の特徴として正しいものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                                                                         | 正誤                          |
| ------ | ------------------------------------------------------------------------------------------------------------ | ----------------------------- |
| ア     | XMLデジタル署名の構文にはCMS（Cryptographic Message Syntax）が用いられる                                     | ×（CMSはPKCS#7で使用）        |
| **イ** | **XML文書中のエレメントに対するデタッチ署名（Detached Signature）を作成し、同じXML文書に含めることができる** | **○**                         |
| ウ     | エンベロービング署名（Enveloping Signature）では、一つの署名対象に複数の署名を付与する                       | ×（意味が異なる）             |
| エ     | 署名対象と署名アルゴリズムはASN.1によって記述する                                                            | ×（XMLではXMLスキーマで記述） |

**解説**
XMLデジタル署名の3つの署名形式：
- **エンベロープ署名（Enveloped Signature）**：署名対象のXML文書の内部に署名を埋め込む
- **エンベロービング署名（Enveloping Signature）**：署名要素の内部に署名対象を含める
- **デタッチ署名（Detached Signature）**：署名対象と署名が別々だが、同一XML文書内に含めることも可能（イが正解）

CMSはS/MIMEやCMSフォーマットで使用され、XMLデジタル署名はW3CのXML Signatureの構文に基づく。ASN.1はCMSで使用される記述言語。

### 問7　SYN/ACKパケット大量観測時の攻撃推定
**正解：ア**

**問題概要**
送信元IPアドレスA、送信元ポート80/tcp、宛先IPが未使用のSYN/ACKパケットが大量観測された場合に推定される攻撃。

**選択肢分析**
| 選択肢 | 内容                                          | 正誤  |
| ------ | --------------------------------------------- | ----- |
| **ア** | **IPアドレスAを攻撃先とするサービス妨害攻撃** | **○** |
| イ     | IPアドレスAを攻撃先とするパスワードリスト攻撃 | ×     |
| ウ     | IPアドレスAを攻撃元とするサービス妨害攻撃     | ×     |
| エ     | IPアドレスAを攻撃元とするパスワードリスト攻撃 | ×     |

**解説**
**SYN反射型DDoS攻撃（SYN Reflection attack）**の典型的なシナリオ：
1. 攻撃者が多数のホストへ、**送信元IPをAに偽装したSYNパケット**（ポート80）を送信
2. 各ホストはSYNパケットを受け取り、送信元（A）へ**SYN/ACKを返送**
3. IPアドレスAは大量のSYN/ACKを受け取り、サービス妨害状態になる

「宛先IPアドレスがホストに割り振られていない未使用のIPアドレス」→ それらのホストはSYNを送ったことが無いため、到着したSYN/ACKは不正なパケットだとわかる。つまりAが被害者（攻撃先）。送信元ポート80は正規のWebサーバに見せかけるための偽装。

### 問8　ドメインフロンティングを悪用した攻撃
**正解：ア**

**問題概要**
ドメインフロンティングを悪用した攻撃の例として正しいものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                                                                 | 正誤                           |
| ------ | ---------------------------------------------------------------------------------------------------- | ------------------------------ |
| **ア** | **マルウェアがCDNの機能を悪用し、実際の通信先を隠蔽して攻撃者のサーバに接続する**                    | **○**                          |
| イ     | マルウェアがDNSキャッシュサーバに偽のDNS情報を蓄積させる                                             | ×（DNSキャッシュポイズニング） |
| ウ     | マルウェアが送信元IPを偽装したDNSリクエストを多数のDNSキャッシュサーバに送り、攻撃対象をダウンさせる | ×（DNS Amplification攻撃）     |
| エ     | マルウェアがドメイン情報管理サイトに不正アクセスし、ドメイン情報を書き換える                         | ×（ドメインハイジャック）      |

**解説**
**ドメインフロンティング（Domain Fronting）**：
- CDN等の大規模インフラを悪用し、**HTTPSのSNI（Server Name Indication）と実際のHostヘッダに異なるドメインを指定**することで、通信先を隠蔽する技術
- 例：TLSのSNIには正規の `cdn.example.com` を指定し、HTTPのHostヘッダには `attacker.evil.com` を指定
- ネットワーク監視ツールはTLSの外側（SNI）しか見えないため、実際の通信先が正規CDNに見える
- C2（コマンド＆コントロール）通信の隠蔽に悪用される

### 問9　情報理論的安全性に基づく暗号技術
**正解：エ**

**問題概要**
情報理論的安全性に基づく暗号技術を選ぶ。

**選択肢分析**
| 選択肢 | 内容                 | 正誤                |
| ------ | -------------------- | ------------------- |
| ア     | DH鍵共有             | ×（計算量的安全性） |
| イ     | RSA暗号              | ×（計算量的安全性） |
| ウ     | 楕円曲線暗号         | ×（計算量的安全性） |
| **エ** | **ワンタイムパッド** | **○**               |

**解説**
暗号の安全性の分類：
- **計算量的安全性**：現実的な計算量では解読できないという前提に立つ安全性。RSA・DH・楕円曲線暗号はいずれも計算量的安全性に依存する（量子コンピュータの脅威あり）。
- **情報理論的安全性（無条件安全性）**：無限の計算能力を持つ攻撃者に対しても解読不可能な安全性。**ワンタイムパッド**（OTP）が唯一の例。鍵長≧平文長かつ鍵を使い捨て・真乱数であることが条件。

### 問11　NIST SP 800-63-3のIdentity Proofing
**正解：ア**

**問題概要**
NIST SP 800-63-3で定義されているIdentity Proofingに該当するものを選ぶ。

**選択肢分析**
| 選択肢 | 内容                                                                                               | 正誤                    |
| ------ | -------------------------------------------------------------------------------------------------- | ----------------------- |
| **ア** | **システムに利用者を登録する際に、利用者本人と写真付き身分証とを照合し、確認する**                 | **○**                   |
| イ     | システムに利用者を登録する際に、希望する利用者IDが既に別の利用者に使用されていないかを確認する     | ×（ID重複チェック）     |
| ウ     | 利用者がシステムにログインした際に、電子メールでログインを利用者本人に確認させる                   | ×（通知・確認プロセス） |

**解説**
NIST SP 800-63-3の3要素：
- **Identity Proofing（本人確認）**：システムに申請者を登録する際、身分証明書等により本人であることを確認するプロセス（アが正解）

- **Federation（連携）**：異なるシステム間でのアイデンティティ情報の連携

IAL（Identity Assurance Level）1〜3で本人確認の厳格さを規定。

**選択肢分析**
| 選択肢 | 内容                                                                                         | 正誤（効果の有無）             |
| ------ | -------------------------------------------------------------------------------------------- | ------------------------------ |
| ア     | 重要操作の都度、利用者のパスワードを入力させる                                               | 効果あり（CSRF対策として有効） |

| 選択肢 | 内容                                                   | 正誤  |
| ------ | ------------------------------------------------------ | ----- |
| ア     | アプリケーションプログラムによるデータ更新命令完了時点 | ×     |

**解説**

### 3.1 重点学習分野

| 優先度 | 分野                         | 理由                                                    |
| ------ | ---------------------------- | ------------------------------------------------------- |

| ★★☆    | 認証・PKI                    | 毎回コンスタントに3問前後                               |
| ★★☆    | Webセキュリティ              | CSRF・XSSの違い等、基礎知識の正確さが問われる           |
| ★★☆    | ネットワーク基礎             | DHCPオプション・OSPFなど基礎問題は確実に正解する        |
| ★☆☆    | 計算・法律・監査             | 各1問、比較的得点しやすい                               |
